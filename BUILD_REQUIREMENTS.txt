# Build Requirements — doorman (lean CPU build)

doorman is a Rust workspace. The default ML backend is ONNX Runtime via the
`ort` crate, which ships a prebuilt CPU library (`download-binaries`), so the
build has NO system ONNX/PyTorch dependency. There is no Python runtime
dependency for inference.

## Ubuntu / Debian system packages

```bash
sudo apt update
sudo apt install -y \
    build-essential \   # C toolchain (linker, headers)
    clang \             # bindgen for native crates
    libclang-dev \      # libclang for bindgen
    libpam0g-dev \      # PAM headers (pam_doorman.so)
    libssl-dev \        # OpenSSL headers (transitive dep of the V4L2/nokhwa camera stack)
    pkg-config \        # locate native libs
    v4l-utils           # V4L2 camera tooling (v4l2-ctl)

# Optional camera backends:
sudo apt install -y \
    libgstreamer1.0-dev libgstreamer-plugins-base1.0-dev \  # camera-gstreamer backend
    ffmpeg                                                  # camera-ffmpeg (CLI webcam path)
```

## Rust toolchain

```bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source "$HOME/.cargo/env"
```

## Build

```bash
# Lean CPU build for real Linux use (daemon + CLI + PAM, native cameras):
make build
#   == cargo build --release -p doormand --no-default-features \
#        --features backend-ort,camera-mock,camera-ffmpeg,camera-v4l2,camera-gstreamer
#      cargo build --release -p pam_doorman

# Default workspace build (dev/preview: CPU ort + mock + ffmpeg camera + preview GUI):
cargo build --release
```

### Feature flags

| Feature             | Meaning                                            | Platform |
|---------------------|----------------------------------------------------|----------|
| backend-ort         | CPU ONNX Runtime (default, bundled lib)            | all      |
| backend-ort-rocm    | AMD ROCm EP (`ort/load-dynamic` + `ort/api-24`)    | Linux    |
| backend-ort-cuda    | NVIDIA CUDA EP (`ort/cuda` + bundled lib)          | Linux    |
| backend-ort-coreml  | Apple CoreML EP (ANE/GPU/CPU, bundled lib)         | macOS    |
| camera-mock         | video-file / synthetic frames (default dev)        | all      |
| camera-ffmpeg       | live webcam via ffmpeg CLI                         | all      |
| camera-v4l2         | V4L2 via nokhwa                                    | Linux    |
| camera-gstreamer    | GStreamer (consumes PipeWire/V4L2 sources)         | Linux    |
| camera-nokhwa       | native webcam (AVFoundation / MSMF / V4L2)         | all      |
| preview             | doorman-preview egui window                        | all      |

## GPU (optional)

```bash
# AMD ROCm (requires a working ROCm install + a ROCm-enabled onnxruntime shared
# lib; backend-ort-rocm uses ort/load-dynamic, so the daemon dlopen()s it at
# runtime via ORT_DYLIB_PATH — nothing GPU is linked at build time):
make build-rocm

# NVIDIA CUDA (requires CUDA + cuDNN; backend-ort-cuda downloads a CUDA-enabled
# ONNX Runtime via ort/download-binaries):
make build-cuda

# Apple CoreML (macOS, Apple Silicon; bundled ONNX Runtime already has CoreML):
cargo build --release --features backend-ort-coreml
```

Note on ONNX Runtime / TLS: doorman uses `ort` 2.0.0-rc.12. rc.12 split TLS out
of `download-binaries`, so any download-binaries path (`backend-ort`,
`backend-ort-cuda`, `backend-ort-coreml`) now pulls in exactly one TLS provider —
we pick `ort/tls-rustls` (ring, no system OpenSSL), keeping the CPU/CUDA/CoreML
builds dependency-free. The `backend-ort-rocm` load-dynamic path links no ONNX
Runtime, so it needs no TLS provider but does add `ort/api-24`. Build the daemon
with ROCm by setting `ORT_DYLIB_PATH` to your ROCm `libonnxruntime.so` at runtime
(see INSTALL.md, AMD ROCm appendix).

Pinned GPU stack (keep in lock-step): `ort` 2.0.0-rc.12 (api-24) <-> ONNX Runtime
1.24.2 (the load-dynamic libonnxruntime.so MUST be this ORT minor) <-> ROCm 7.2.4
for gfx1103 (Radeon 780M). A mismatched ORT minor breaks the dlopen at runtime.

## macOS (dev only)

`cargo build` works on macOS for development and preview. The PAM module compiles
to an empty cdylib there (no libpam); the login/lock-screen unlock path is
Linux-only and is validated on Ubuntu.
